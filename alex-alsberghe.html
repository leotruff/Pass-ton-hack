Problematique 1 - Identifier un service reseau vulnerable

Enonce / Descriptif

Contexte : un challenge reseau expose plusieurs ports sans indication. Objectif : identifier rapidement le service exploitable sans utiliser de donnees sensibles.

Resolution

  1. Scan initial des ports ouverts pour obtenir la surface d'attaque.
  2. Enumeration des versions de services detectes.
  3. Test controle des reponses HTTP pour isoler l'entree faible.
  4. Validation de l'hypothese avec une preuve reproductible.

Outil(s) / Ressource(s) utilisees

  • Nmap
  • Curl
  • Documentation CVE publique

Difficulte(s) rencontree(s) + correction

Difficulte : trop de ports bruyants au premier scan. Correction : filtrage progressif et priorisation des services web.

Ce que j'ai appris

  • Interpreter les retours de scan sans se perdre dans les faux positifs.
  • Relier une version de service a une faille documentee.
  • Documenter une preuve reseau de facon claire et defendable.

Grand Oral : ce que je peux reutiliser

Je peux reutiliser cette demarche pour expliquer la priorisation technique et la methode d'investigation en cybersecurite.

Commandes utilisees

# Reconnaissance initiale
nmap -sV cible.example
curl -I http://cible.example
nmap --script vuln cible.example

Problematique 2 - Stabiliser une resolution de challenge reseau

Enonce / Descriptif

Contexte : la premiere solution fonctionnait de maniere intermittente. Objectif : obtenir une methode stable, repetable et presentable.

Resolution

  1. Reproduction du probleme sur plusieurs tentatives.
  2. Comparaison des paquets et temps de reponse.
  3. Ajustement des parametres de requete et du timing.
  4. Validation finale sur une sequence complete.

Outil(s) / Ressource(s) utilisees

  • Wireshark
  • Curl en mode verbeux
  • Notes de tests comparatifs

Difficulte(s) rencontree(s) + correction

Difficulte : reponses variables selon la charge. Correction : protocole de test reproductible avec fenetres horaires et options stables.

Ce que j'ai appris

  • Mesurer avant de corriger evite les changements inutiles.
  • La stabilite d'une solution compte autant que sa reussite ponctuelle.
  • La preuve par comparaison est tres convaincante a l'oral.

Grand Oral : ce que je peux reutiliser

Je peux reutiliser ce cas pour montrer ma capacite a diagnostiquer une instabilite et a fiabiliser une solution technique.

Commandes utilisees

# Analyse de stabilite
curl -v http://cible.example/endpoint
tcpdump -i eth0 host cible.example
nmap -Pn -p 80,443 cible.example
